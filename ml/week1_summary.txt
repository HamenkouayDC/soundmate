Дата фиксации: 23.06.2026

ЗАДАЧА 1: Протестировать Spotify API и Last.fm API
Статус: ЧАСТИЧНО ВЫПОЛНЕНО

- Spotify: код написан и готов (spotify_client.py), но НЕ протестирован
  на реальных данных. Причина: с февраля 2026 Spotify требует Premium-
  подписку у владельца приложения для доступа к Web API (Development
  Mode). У команды Premium ни у кого нет. Это внешнее ограничение
  платформы, не недоработка разработки.

- Last.fm: полностью протестирован на реальных данных. Создан тестовый
  аккаунт, подключён Spotify для автоматического скробблинга, получена
  реальная история прослушиваний (Kiss, Twisted Sister, Limp Bizkit).
  fetch_top_artists, fetch_top_tracks, fetch_artist_tags — все три
  функции проверены и возвращают корректные данные.

ЗАДАЧА 2: Подать заявку на SoundCloud API
Статус: ЗАБЛОКИРОВАНО

SoundCloud изменил процесс доступа к API: вместо формы заявки с
ожиданием подтверждения теперь требуется платная подписка SoundCloud
Artist Pro для регистрации приложения. У команды подписки нет, действие
невозможно до её появления. Зафиксировано как риск (см.
soundcloud_application.txt) — это уже второй случай в спринте, когда
музыкальная платформа вводит платный барьер для доступа разработчиков
(первый — Spotify Premium, см. задачу 1).

ЗАДАЧА 3: Спроектировать структуру музыкального эмбеддинга
Статус: ВЫПОЛНЕНО

Финальная структура: вектор длины 25 (5 аудио-признаков + 20 жанровых
долей), все компоненты в диапазоне [0, 1].

Решённые вопросы:
- indie — модификатор, не отдельная категория (genre_categories.py)
- Финальный список из 20 жанровых категорий зафиксирован
- Нормализация всех 25 компонент в [0, 1] для будущего Faiss

Проверено на реальных данных пользователя (не только на синтетике):
- Жанровая часть посчитана из реальных Last.fm-тегов (Kiss, Twisted
  Sister) — результат: genre_rock=0.4, genre_metal=0.2, genre_other=0.4,
  остальные категории = 0. Совпадает с ручным расчётом по логике
  голосования в _genre_distribution.
- Аудио-часть на синтетических данных (так как Spotify недоступен)
  показывает корректное усреднение и нормализацию tempo.
- Оба режима работы (с аудио-данными и без, через NaN) подтверждены
  рабочими.

Известное ограничение, зафиксированное для будущей доработки:
- Жанровый профиль строится на уровне артиста (artist.getTopTags), а
  не на уровне конкретного трека (track.getTopTags). Это усредняет
  жанр по всей дискографии артиста и не различает, например, более
  тяжёлые и более мелодичные периоды творчества одного исполнителя.
  Точное решение требует кратного роста числа API-запросов (на каждый
  трек отдельно, а не на каждого артиста) — отложено на следующие недели.

ОБЩИЙ РИСК ДЛЯ ОБСУЖДЕНИЯ С КУРАТОРОМ:

Две из трёх целевых музыкальных платформ (Spotify, SoundCloud) ввели
платные барьеры для доступа разработчиков в течение одного спринта.
Если эта тенденция продолжится (Яндекс Музыка — следующая в списке
интеграций проекта), бюджет проекта должен явно учитывать подписки на
профильные сервисы как необходимое условие работы, а не считать доступ
к музыкальным API бесплатным по умолчанию.
